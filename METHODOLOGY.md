## 1. Corpus construction (`keyword_search/`)

- **Source**: 295 German state-curriculum PDFs (Lehrpläne, Bildungsstandards, EPA) across 5 subjects (Biologie, Ethik/Philosophie, Sozialkunde, Geschichte, Psychologie) and all 16 Bundesländer + national/KMK documents.
- **Extraction**: `pymupdf`, page-by-page, header/footer clipped (72pt), text normalized (dash unification + NFKC), then whitelist-cleaned to `[a-zA-ZäöüÄÖÜßéèêëçñ\s,.;:/-]`.
- **Matching**: regex word-boundary search per concept keyword variant (case-insensitive, except "Gen" which is case-sensitive to avoid matching "gegen"), ±30-word context window per match → one excerpt row per match.
- **Known limitation (found 2026-07-21)**: one document, `LP RheinPfalz Philosohpie Gym Sek2.pdf`, has a broken/non-standard embedded font encoding. `pymupdf` mis-maps German special characters: ü→Ÿ (U+0178, 544×), ä→Š (U+0160, 463×), ö→š (U+0161, 138×), ß→ã (U+00E3, 42×). The whitelist-cleaning step then silently *deletes* these mis-mapped characters (they're not in the whitelist) rather than preserving or flagging them, which is how "Schüler" became "Schler" in ~57 excerpts feeding into both downstream pipelines. Verified to be isolated to this single document (scanned all 295 raw `.txt` mirrors for the telltale characters — zero other matches). Fix: character remap (Ÿ→ü, Š→ä, š→ö, ã→ß) applied to this document's already-extracted text, preferred over re-extraction/OCR since **the original PDF is not currently available on this machine** (see §5 — provenance gap, needs resolution before the fix can be finalized as re-extraction; remapping the existing extracted text does not require the source PDF).
- **Pipeline fix (implemented 2026-07-21, on branch `data-rerun-2026-07`)**: `remap_font_encoding_artifacts()` added to `keyword_search/src/utils.py` (Ÿ→ü, Š→ä, š→ö, ã→ß), applied in both extraction paths — `normalize_text()` in `keyword_search.py` (feeds `results.csv`) and `pdf_dir_to_txt()` in `utils.py` (feeds the `docs/data/txtfiles/` "Volltext" mirror) — so both are fixed consistently. Verified against the actual affected document: "europŠische"→"europäische", "fachŸbergreifenden"→"fachübergreifenden", zero Ÿ/Š/š/ã characters remain after remap. `clean_hyphenated_text()` also now reports (via `print()`) any alphabetic character it's about to strip that isn't a recognized German/Latin letter, so a future document with an undiscovered version of this issue is caught by inspection instead of by accident.

**Regeneration — executed 2026-07-21** via `scripts/fix_rheinpfalz_philosophie_corruption.py`. Does not require the source PDF: the corrupted characters were still present (mis-mapped but not yet stripped) in the existing raw `.txt` mirror, so re-running the match/context-window extraction against that corrected text and splicing the result back into `results.csv` was sufficient — sidesteps the PDF-backup dependency in section 8 entirely for this fix. 350 rows regenerated (vs. 349 corrupted originals — the small difference is the documented tradeoff of using the unclipped raw mirror instead of a true per-page-clipped PDF re-extraction: a handful of excerpts near page boundaries may include a stray header/footer word). Verified: zero mis-mapped characters remain in the regenerated rows, and a corpus-wide re-scan confirms zero remaining corrupted rows anywhere in `results.csv`. Written to both `keyword_search/out/results.csv` and `docs/data/results.csv` (35,894 total rows, was 35,893).

**Also found and fixed while implementing this**: `clean_hyphenated_text()`'s new defensive stripped-character warning (above) crashed with `UnicodeEncodeError` on Windows consoles when the stripped character fell outside cp1252 entirely (this document also contains legitimate Greek philosophical terms/symbols — χ, η, α, ρ, τ, ε, ν — which the warning correctly flags as "unexpected," but printing them crashed the run rather than just warning). Fixed with an encode/decode-with-replace fallback so the warning degrades gracefully instead of taking down the pipeline.

## 1.5 Encoding integrity: a first-class concern for German-language CCS, not a footnote

This project has now hit encoding problems twice, at two completely different layers of the stack, and both are worth recording — not just as bugs fixed, but because **diacritic-dependent languages make silent encoding corruption a data-integrity risk in a way that ASCII-only corpora simply don't face**, and that has direct implications for anyone doing German-language (or French, Scandinavian, Central/Eastern European…) Computational Curriculum Studies.

**Incident 1 (§1, 2026-07-21)**: PDF font-encoding mis-mapping at the *extraction* layer. `pymupdf`'s glyph mapping for one document's embedded font was broken, so ü/ä/ö/ß came out as Ÿ/Š/š/ã in the raw extracted text — valid-looking characters, not obviously broken, that then got silently *deleted* by a downstream whitelist-cleaning regex that didn't recognize them. Net effect: "Schüler" quietly became "Schler" in ~57 excerpts, with no crash, no warning, nothing to notice unless someone specifically went looking.

**Incident 2 (§4.4/§4.6, 2026-07-22)**: `sys.stdout` encoding mismatch at the *console/log* layer. This Windows machine's Python defaults `sys.stdout` to `cp1252` (confirmed via `sys.stdout.encoding`; no `PYTHONIOENCODING`/`PYTHONUTF8` set). Any `print()`/`logging.StreamHandler(sys.stdout)` output containing umlauts, when redirected to a file via shell `>` for background-job logging (as done throughout this session's production runs), gets silently mis-encoded as single-byte cp1252 instead of two-byte UTF-8 — again, no crash, no warning, just wrong bytes sitting in a file that looks fine until decoded strictly.

**Why this is a bigger deal for German specifically than it would be for an English-language corpus**: English text can hit similar issues, but the practical blast radius is usually a handful of loanwords or proper nouns (café, naïve, Müller-as-a-surname) — an edge case you can often get away with ignoring. German orthography uses umlauts and ß *pervasively*, including in extremely high-frequency, semantically load-bearing vocabulary — and concretely, in this project's own domain vocabulary: one of the 22 core concepts studied here is literally *Bedürfnis*. A silent encoding corruption in a German corpus isn't a rare edge case touching a few proper nouns; it has near-total surface area, because almost any sufficiently large chunk of German prose contains multiple affected characters. And because a topic-modelling/keyword-search pipeline works by *exact token matching* (stopword lists, keyword search terms, term-frequency counts, coherence scoring), a single wrong byte doesn't just look ugly — it silently changes a word's identity for every counting/matching/filtering step downstream. A stopword list entry "für" will not match a corrupted "f?r" token; the function word leaks into the topic model uncaught. This is mechanistically the *same* failure mode as Incident 1, just triggered by a different bug.

**What actually protected this project from Incident 2 causing real damage**: explicit, deliberate `encoding="utf-8"` on every file-writing call that produces actual data (`json.dump` targets, the pipeline's own `FileHandler`) — verified this session by strict byte-level `.decode("utf-8")` tests against every CSV/JSON output for the four concepts most likely to expose the problem (Bedürfnis, Gefühl, Glück, Rationalität), plus a hex-level check of filenames on disk. Pandas' `to_csv()` was also confirmed (via source inspection, pandas 2.0.1) to default to UTF-8 regardless of OS locale, not the OS-locale-dependent behavior Python's bare `open()`/`print()` exhibit. The only files actually affected were ad-hoc shell-redirected console-log captures (`production_bertopic.log`, `full_sweep.log`, `timing_pilot.log`) — human-readable diagnostics, never a data source for anything downstream.

**The methodological lesson, generalized**: for a diacritic-dependent-language CCS pipeline, encoding correctness cannot be assumed from "the code looks reasonable" or "the terminal output looks fine" — both directions of that intuition failed at different points in this project. Incident 1 looked fine in the terminal and *was* actually corrupted. Incident 2 looked corrupted in the terminal and *was not* actually corrupted (a display artifact from redirecting a cp1252-default stdout, not a property of the underlying string). **Visual inspection of German text for encoding correctness is not reliable in either direction — only a programmatic round-trip/decode check is.** Concrete practices adopted going forward in this repo, worth stating explicitly for anyone replicating this pipeline on another corpus/language:
- Every file-writing call that produces actual analysis data specifies `encoding="utf-8"` explicitly — never rely on a platform default.
- `PYTHONIOENCODING=utf-8` (and `PYTHONUTF8=1`) set explicitly in the environment for any subprocess launch whose console output will be captured to a log file, even though this only affects log readability, not data — worth doing on the first pass rather than discovering it mid-project.
- When verifying encoding correctness (e.g., after a corpus-affecting change), verify via `raw_bytes.decode("utf-8")` succeeding/failing, not via reading a terminal or log rendering — this session's own diagnostic scripts fell into the exact same stdout-encoding trap while investigating the problem, and would have produced a false "yes it's corrupted" conclusion if the byte-level check hadn't been done as the actual arbiter.
- For a book/dataset publication specifically: state the verification method as an explicit data-quality claim ("all corpus/output files verified as valid UTF-8 via strict decode test on `<date>`") rather than an unstated assumption — this is exactly the kind of infrastructure-level methodological choice that's invisible in results but load-bearing for their validity, and belongs in a methods section for a project working with a diacritic-heavy language.

## 2. Stopword strategy

Audited corpus-wide term frequency (35,893 excerpts, 2,009,632 token occurrences, 45,611 distinct tokens) against both pipelines' existing stopword lists, which were found to diverge significantly before this rerun (BERTopic: NLTK German ~230 words + 15 ad-hoc extras; LDA: spaCy German ~543 words + a different 26 ad-hoc extras — near-zero overlap between the two extras sets).

**Decision (approved 2026-07-21): one shared stopword list for both pipelines**, three tiers:

- **Tier 1 — global administrative/pedagogical stopwords** (added): curriculum-document scaffolding — `schülerinnen`, `schüler`, `kompetenz`, `kompetenzen`, `kompetenzerwartungen`, `sachkompetenz`, `urteilskompetenz`, `methodenkompetenz`, `handlungskompetenz`, `unterricht`, `fach`, `themen`, `hinweise`, `bildung`, `grundlage`, `grundlagen`, `aspekte`, `bezüge`, `lernenden`; plus Bildungsstandards competency-verbs ("Operatoren"): `erläutern`, `beschreiben`, `beurteilen`, `reflektieren`, `analysieren`, `erklären`, `bewerten`, `erkennen`, `diskutieren`, `vergleichen`, `unterscheiden`, `darstellen`, `kennen`. Evidence: the "Kompetenz" word family alone totals 10,467 occurrences, filtered by neither pipeline previously.
- **Tier 2 — explicitly protected from stopwording**, despite high frequency, because they are target concepts or closely adjacent substantive content: `menschen`/`mensch`, `handeln`/`handelns`/`handlungen`, `normen`, `werte`/`werten`, `freiheit`, `gerechtigkeit`, `verhalten`, `evolution`, `glück`, `kultur`, `moral`/`moralische`, `demokratie`, `gesellschaft`, `ethik`, `welt`, `natur`, `geschichte`, `zukunft`, `verantwortung`, `entwicklung`, `lernen` (ambiguous — resolved to keep). Rationale: a naive "stopword everything frequent and uncovered" pass would have deleted the actual subject matter of the study.
- **Tier 3 — per-concept self-seed exclusion** (not global): each concept's own keyword variants (from `keywords.txt`) are excluded only from that concept's own topic vectorizer, since by construction ~100% of a concept's documents contain its own seed term — e.g. strip "freiheit"/"autonomie" from the Freiheit model only, while leaving "freiheit" as ordinary vocabulary in every other concept's model (where it may be genuinely informative).

All three tiers approved 2026-07-21. **Implemented** same day as `shared_stopwords.py` (repo root) — `stopwords_for_concept(concept)` returns the unified base list plus that concept's Tier 3 self-seed exclusions; both `bertopic_pipeline_v2.py` and `lda_topic_model.py` now import from it instead of maintaining separate lists. On branch `data-rerun-2026-07`.

## 3. Sentence-embedding model (BERTopic)

**Original**: `all-MiniLM-L6-v2` — despite the pipeline's own comment describing it as "multilingual-capable," this model is trained overwhelmingly on English sentence pairs and is not part of the sentence-transformers multilingual family.

**Diagnostic pilot (2026-07-21)**: encoded Verhalten's full corpus (1,155 excerpts — the concept Susan flagged as suspiciously under-differentiated) with both the original model and a genuine multilingual candidate, then fit identical UMAP(n_neighbors=15, n_components=5, min_dist=0.0, cosine) + HDBSCAN(min_cluster_size=15, min_samples=5, eom) on each, seed=42:

| Model | Natural clusters found | Outlier rate | Silhouette (non-outlier) |
|---|---|---|---|
| `all-MiniLM-L6-v2` | 4 | 0.0% | 0.7875 |
| `paraphrase-multilingual-MiniLM-L12-v2` | 23 | 19.0% | 0.5477 |

**Interpretation**: 4 clusters with literally zero outliers across 1,155 documents is a red flag, not a clean corpus — it indicates the embedding space is too homogeneous to let HDBSCAN find real structure. The multilingual model's 23 clusters + a plausible ~19% outlier rate is the expected signature of an embedding space that can actually differentiate content. The silhouette drop (0.79→0.55) is not a quality regression: a very high silhouette on 4 giant undifferentiated clusters mostly reflects trivial within-blob proximity, whereas 0.55 across 23 genuinely distinct clusters reflects real (and still solidly separated — above the ~0.5 "reasonable structure" threshold) structure.

**Decision (approved 2026-07-21)**: adopt `paraphrase-multilingual-MiniLM-L12-v2` for the full rerun. **Implemented** same day in `bertopic_pipeline_v2.py` (`EMBEDDING_MODEL_NAME`) on branch `data-rerun-2026-07`.

**Mechanistic note for the record**: stopword filtering does *not* reach BERTopic's clustering step — HDBSCAN clusters on raw SentenceTransformer embeddings of the *unfiltered* excerpt text; stopwords only shape the post-hoc c-TF-IDF topic-label representation. This means the embedding-model choice is the dominant lever for "content vs. context" differentiation, more fundamental than stopword tuning. See the flagship-concept deep-dive plan (§4) for a further test of context-stripping *before* embedding.

## 4. Topic-count strategy

**Problem**: concept corpus sizes span an 18× range (267 excerpts for Emotion to 4,852 for Mensch). A single fixed `nr_topics` for every concept is statistically indefensible — it over-fragments small corpora and over-merges large ones.

**Decision (approved 2026-07-21)**: run a systematic topic-count sweep (K ∈ {5,7,10,12,15,20}) across **all 22 concepts**, both BERTopic (cheap via `reduce_topics()` re-merging a single base fit, no need to refit embeddings/UMAP/HDBSCAN per K) and LDA (existing 5/7/10 extended to the same range), with:
- Topic coherence (c_v/c_npmi) per K per method, to find the diminishing-returns point.
- Term-overlap/agreement scoring between BERTopic and LDA topics at matched K — convergence between two structurally different methods is evidence of real signal; divergence flags a concept for cautious interpretation.
- Topic-size distribution per K, to catch degenerate splits.

Applies to all 22 concepts, not just the 3 flagship ones — chosen over the cheaper "flagship-only" alternative for book-wide methodological rigor.

**Implemented 2026-07-21**: `DEFAULT_NR_TOPICS_CEILING = 11` in `bertopic_pipeline_v2.py`, now the default (was `None`/uncapped). Confirmed against BERTopic 0.16.4's own source (`_reduce_topics`/`_reduce_to_n_topics`) that `nr_topics` counts the outlier bucket toward its total when outliers exist (`AgglomerativeClustering(n=nr_topics - self._outliers, ...)`) and only ever merges DOWN when the natural topic count exceeds it — so 11 gives "up to 10 real topics + 1 outlier," and a concept that naturally settles below that is left alone, exactly the "ceiling not target" behavior Strategy B calls for, with no additional custom logic needed. `topic_sweep.py` (new, repo root) implements the sweep + coherence + agreement infrastructure: per concept, one natural BERTopic fit is cheaply re-merged to each K via `reduce_topics()` (reuses the cached c-TF-IDF hierarchy), LDA is refit from scratch at each K (no equivalent cheap reduction exists for LDA), gensim `c_v` coherence is computed for both against the concept's own tokenized corpus, and a Hungarian-matching (`scipy.optimize.linear_sum_assignment`) average-Jaccard agreement score is computed between the two methods' topic term sets at each K. Smoke-tested end-to-end on Vorurteil (285 docs, K=5 and 7) — mechanically correct, produced plausible coherence (~0.39-0.45 c_v) and a notably low agreement score (~0.06-0.07) worth re-examining once the full sweep runs (small-corpus, low-K result, not necessarily representative). **Not yet run across all 22 concepts** — that's the Phase 3 execution step.

**Also found and fixed while building this**: `compute_or_load_embeddings()`'s cache filename carried no embedding-model identity, so the 2026-07-21 model swap (this section, above) would have silently reused the stale `all-MiniLM-L6-v2` embeddings cache instead of recomputing under the new model — a serious latent correctness bug that would have quietly invalidated the whole rationale for switching models. Fixed via a new `embedding_model_slug()` helper, applied to every embedding-derived cache path: the global embeddings cache, per-concept BERTopic model pickles, per-concept UMAP projections, and the global UMAP projection in `build_global_projection.py`.

### 4.1 Timing pilot (2026-07-21/22)

Before committing to the full 22-concept sweep, ran `topic_sweep.py --concepts Vorurteil,Kultur,Mensch` — smallest (285 excerpts), mid-size (1,466), and largest (4,852) concept corpora, chosen to bound the cost estimate — across the full K∈{5,7,10,12,15,20} × {BERTopic, LDA} grid.

**Timing**:
- One-time full-corpus embedding pass (all 26,165 excerpts across all 22 concepts; shared/cached, not per-concept): 16m53s. Written to `bertopic/src/cache/embeddings_all__paraphrase-multilingual-MiniLM-L12-v2.npy`; reused by every subsequent run at no extra cost as long as corpus and model stay fixed.
- K-sweep + coherence + agreement scoring for the 3 pilot concepts (6,603 excerpts combined): 61m44s.
- Linear-by-document-count extrapolation to the full 26,165-excerpt corpus: **≈4 hours** for the complete 22-concept diagnostic sweep. Order-of-magnitude only — LDA/BERTopic cost isn't perfectly linear in corpus size (spaCy lemmatization is ~linear; sklearn LDA and UMAP/HDBSCAN have some superlinear components) — but sufficient to greenlight running the full sweep unattended rather than requiring a second, larger pilot.

**Coherence/agreement findings from the pilot** (shapes how the full-sweep results should be read, not just the timing decision):
- BERTopic c_v coherence rises with K and generally exceeds LDA's at matched K (e.g. Mensch: BERTopic 0.47→0.71 across K=5→20 vs. LDA's flat 0.36–0.43 over the same range).
- BERTopic's *real* topic count (post `reduce_topics()`) plateaus below the requested K once a corpus's natural cluster structure is exhausted — Vorurteil (smallest, 285 docs) plateaus at 10 real topics from K=10 onward, i.e. asking for 15 or 20 doesn't manufacture more topics there. Larger corpora (Mensch, Kultur) keep producing more real topics through K=20 without plateauing yet, meaning their coherence-vs-K curve may not have peaked within the tested range — worth weighing when choosing a final per-concept K for large concepts specifically.
- BERTopic/LDA term-agreement (Hungarian-Jaccard) is low across the board in the pilot (0.04–0.13, no clear trend with K). Read as expected structural difference between the two methods (BERTopic clusters embedding-space geometry; LDA models token co-occurrence) rather than evidence either method is "wrong" — supports treating the two as complementary triangulation rather than picking one as ground truth, consistent with this section's original rationale for running both.

**Decision**: pilot confirmed the sweep infrastructure is mechanically correct and the ~4-hour full-sweep cost is acceptable; proceeded directly to the full 22-concept run rather than a second, larger pilot.

### 4.2 Full 22-concept sweep — launch note (reproducibility)

First launch attempt (`nohup … &` inside a tool call that already had its own background-execution flag set) produced a false "completed, exit 0" signal within about a minute, with a 0-byte log and no `topic_sweep_report.json` produced — the actual python process had been silently orphaned/killed rather than run to completion, because shell-level backgrounding (`&`) was layered on top of the tool's own process tracking instead of left to it alone. **Caught before being mistaken for a real (and wrong) result** by checking the log file size and the report file's existence rather than trusting the "completed" notification in isolation. Relaunched using only the tool's native background execution, no shell-level `&`/`nohup`; confirmed genuinely alive by watching the log grow across two follow-up checks (0 → 1,216 bytes of real joblib worker-spawn output) before treating it as running.

Recorded here because the same launch pattern will recur for the Phase 3 production BERTopic/LDA reruns: **verify a background job is actually alive via artifact growth (log size, output files), not just the completion notification**, whenever shell-level and tool-level backgrounding could plausibly conflict.

### 4.3 Full 22-concept sweep — mid-run interruption and checkpointing fix (2026-07-22)

The correctly-relaunched sweep (§4.2) ran for roughly 3 hours and progressed through 10 of 22 concepts (Anpassung → Glück, in `bertopic_pipeline_v2.DEFAULT_CONCEPTS` order) before the background process died with no error recorded — the tool-level task tracker itself reported the job as belonging to "a previous session" with no completion record, consistent with an external interruption (session/process restart) rather than a bug in the script or the relaunch. Per-concept BERTopic base fits for that batch were confirmed still cached on disk (e.g. `bertopic/src/cache/Glück/topic_model_Glück__paraphrase-multilingual-MiniLM-L12-v2.pkl`, written at the time the process died) — so the expensive embedding+UMAP+HDBSCAN step for those 10 concepts is not lost.

**However**, `topic_sweep.py`'s original design (`run_full_sweep()`/`main()`) only wrote `topic_sweep_report.json` once, after every concept in the run finished — so all ~3 hours of coherence/agreement numbers for those 10 concepts were lost with the process, with nothing recoverable from the report side despite the BERTopic fits surviving.

**Fix (implemented same day, before relaunching a second time)**: `run_full_sweep()` now writes the accumulated report (via an atomic tmp-file-then-`replace()`) after *every* concept, not just at the end, and `main()` gained resume support — on start, it loads any existing `--out` file and skips concepts already present in it (a `--no-resume` flag forces a full re-sweep if ever needed). This means a future interruption loses at most one concept's worth of sweep time, and a routine relaunch after any interruption is a no-op for already-completed concepts rather than a full restart. This does not help recover the specific 10 concepts lost in this incident (nothing was ever written for them), but does mean they will not need to be redone if *this* relaunch is also interrupted.

**Status**: sweep relaunched 2026-07-22 (all 22 concepts requested; resume found no existing report, so effectively a fresh run) with checkpointing active, writing incrementally to `bertopic/src/out/topic_sweep_report.json`. Completed cleanly this time — all 22 concepts, no errors.

### 4.4 Full sweep results and the production topic-count decision (2026-07-22)

**The 5–20 grid alone was insufficient to read off a "right" K per concept.** 13 of 22 concepts had not plateaued in BERTopic real-topic-count by K=20 (coherence and real-topic-count both still rising), so extrapolating a final ceiling from that grid the way the pilot's 3-concept read suggested (§4.1) would have undercounted badly for exactly the concepts that most need richer differentiation.

**Better source of truth, already on disk at no extra cost**: `sweep_concept()` fits each concept's BERTopic base model *uncapped* (`nr_topics=None`) before cheaply re-merging it down to each tested K via `reduce_topics()` — so the cached model pickle for every concept already holds its true natural (unconstrained) topic count. Read directly from `bertopic/src/cache/{concept}/topic_model_{concept}__paraphrase-multilingual-MiniLM-L12-v2.pkl`:

| Handeln | 67 | 21.2% |
| Kultur | 38 | 19.9% |
| Wert | 38 | 23.7% |

| Ursache | 28 | 22.0% |

**This directly confirms and generalizes the original concern.** The prior production default (`DEFAULT_NR_TOPICS_CEILING = 11`, i.e. up to 10 real topics) was crushing nearly every concept above ~1,000 documents to roughly a third or less of its natural richness — not just Mensch/Verhalten/Evolution (the three originally flagged), but Handeln, Kultur, Wert, Gerechtigkeit, Ursache, Moral, Freiheit, and Norm too. Small concepts were unaffected (a ceiling only ever merges *down*, never invents topics), which is why the problem hadn't been visible from those corpora.

**Decision**: raise `DEFAULT_NR_TOPICS_CEILING` from 11 to **31** (up to 30 real topics + 1 outlier bucket). Checked against the app's BERTopic legend panel (`docs/styles.css`: `.legend-panel { max-height: 640px; overflow-y: auto; }`) — it scrolls, so it isn't a hard visual ceiling; 30 was chosen as a practical bound that still meaningfully reins in the two extreme outliers (Mensch 90→30, Handeln 67→30) and mildly trims Kultur/Wert/Gerechtigkeit (38/38/33→30), while leaving all other 17 concepts — **including both remaining flagship concepts, Verhalten (22) and Evolution (22)** — fully uncapped at their natural count. This is a direct, evidence-based resolution of the original ask.

**Known gotcha, verified before acting on it**: `fit_or_load_bertopic()`'s cache key (`topic_model_{concept}__{embedding_slug}.pkl`) does **not** include `nr_topics` — unlike the embedding-model slug (§4, earlier fix), the ceiling value is applied only at fit time and is invisible to the cache lookup. Since the sweep populated this exact cache with `nr_topics=None` (fully uncapped) fits for all 22 concepts, a normal production run (`force_topics=False`, the default) would silently return those uncapped models regardless of the new ceiling — Mensch would ship with all 90 topics, not 30. **`--force-topics` is therefore required for this production run** to force a genuine refit under the new ceiling; noted here so this doesn't get missed on a future rerun where the cache is in a similar state.

**LDA companion decision**: the app's LDA page exposes topic count as a **live user-facing dropdown** (`docs/lda.html` `#lda-k`, backed by `N_TOPICS_LIST` in both `lda_topic_modelling/src/lda_topic_model.py` and `scripts/build_docs_data.py`, currently `[5, 7, 10]` in both, applied identically to every concept — not a per-concept backend choice like BERTopic's). Rather than redesign that existing exploratory feature, `N_TOPICS_LIST` is widened to `[5, 7, 10, 12, 15, 20]` — the exact grid this sweep validated, so every option the dropdown offers now has a documented coherence number behind it, and readers can explore granularity themselves across a much wider range than before (previously capped at 10, matching the same undercounting problem as BERTopic's old ceiling).

### 4.6 Critical, pre-existing bug: BERTopic was silently ASCII-stripping every topic label (2026-07-23)

**Found while re-verifying "everything" per a user request to restart/re-test, not something introduced this session** — but it predates this session and affects every BERTopic topic label/term this project has ever produced.

**Symptom**: a freshly-run comparison script's topic-term output contained words like "abhngigkeit", "gefhle", "franzsische" — umlauts simply missing, not mis-encoded. Verified at the codepoint level (not a terminal-display artifact, unlike §1.5's earlier finding) before treating it as real: the string genuinely contained no `ä`/`ü`/`ö` character at all. Confirmed the *same* pattern already present in the just-shipped production data (`bertopic/src/data/Mensch.json`: "Glckliches", "Gefhle", "Abhngigkeit" as topic-label substrings) — meaning this was not new, just newly noticed.

**Root cause, isolated via a minimal reproduction**: BERTopic's own `_preprocess_text()` (`bertopic/_bertopic.py`) runs `re.sub(r"[^A-Za-z0-9 ]+", "", doc)` — an ASCII-only strip — on every document used to build a topic's c-TF-IDF representation, but *only* `if self.language == "english"`. `BERTopic.__init__` sets `self.language = language if not embedding_model else None`, defaulting `language` itself to `"english"` if never passed. `bertopic_pipeline_v2.py`'s `_build_model()` (used by both the production pipeline and `topic_sweep.py`, which imports the same function) passes neither `language=` nor `embedding_model=` — embeddings are computed separately and injected via `fit_transform(docs, embeddings)` — so `self.language` was silently `"english"` this entire time, and every topic's underlying text was ASCII-stripped before term extraction. Confirmed with a controlled before/after repro (identical synthetic German sentences, `language="multilingual"` vs. unset) — before: `"abhngigkeit"`, `"gefhle"`, `"franzsische revolution"`, `"prgen"`; after: `"abhängigkeit"`, `"gefühle"`, `"französische revolution"`, `"prägen"`.

**Fix**: `_build_model()` now passes `language="multilingual"` explicitly (`bertopic_pipeline_v2.py`, semantically accurate given `EMBEDDING_MODEL_NAME`). Verified LDA is *not* affected — spot-checked `docs/data/lda/Mensch/10/topic_terms_Mensch_10.csv` directly (not via a print statement, learning from §1.5) and found correctly-formed umlaut terms throughout ("ausgewählt", "begründen", "ernährung", "prägen", "ökologisch") — LDA uses an entirely separate spaCy/sklearn codebase with no equivalent hardcoded-language preprocessing step.

**Scope of what needed redoing**: only BERTopic's topic *labels and terms* were affected — clustering itself operates on the precomputed embeddings (unaffected, computed from the correctly-UTF-8 source text), so **the natural per-concept topic counts underlying the §4.4 ceiling decision (2–90 range, ceiling=31) are unaffected and remain valid** — no need to revisit that decision. What *did* need redoing: the full production BERTopic rerun (`--force-topics`, all 22 concepts, ~2 minutes this time since UMAP/embeddings were already cached — only the fit+label step reran), the global-projection re-merge, the `docs/data` mirror rebuild, and the flagship content-context pilot (§5) — its own cache (`bertopic/src/cache_content_focused/`) held pre-fix models and was deleted before rerunning, since `fit_or_load_bertopic()`'s cache key doesn't encode `language` either (the *third* instance of this general class of gotcha this session, after the embedding-model slug and `nr_topics` — see §4.4's "Known gotcha").

**Not yet redone, flagged rather than silently ignored**: `topic_sweep.py`'s K-sweep (§4.1–4.4, `topic_sweep_report.json`) also used the buggy `fit_or_load_bertopic()` throughout, meaning `bertopic_coherence_cv` and `agreement_jaccard` were computed against ASCII-stripped BERTopic terms — coherence lookups for umlaut-bearing terms against the (correctly-tokenized) corpus would have under-matched, plausibly **deflating BERTopic's coherence scores and BERTopic/LDA agreement scores specifically**, for any topic whose salient vocabulary includes umlauts (i.e., most of them, given German). The topic-*count* data driving the ceiling decision is unaffected (see above), but the coherence/agreement *numbers* in §4.1/§4.4/§4.5 should be treated as a plausible lower bound for BERTopic, not a fully reliable reading, until the sweep is rerun under the fix — a multi-hour job not undertaken in this pass given the production dataset itself is what needed to be correct by morning. Recommend re-running `topic_sweep.py --concepts all` under the fix before treating the coherence/agreement comparison as final for the book's methods section.

## 5. Flagship deep-dive: Verhalten, Mensch, Evolution
